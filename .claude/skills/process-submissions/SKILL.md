# 中文独立开发者 Vibe Coding 作品集：自主处理投稿

你的任务：处理 `DevEloLin/chinese-indie-vibe-coding` 的投稿——带 `投稿` 标签的 Issue（来自投稿表单）
与开放的 PR。有效作品加进 `README.md` 的作品清单，感谢提交者并关闭；垃圾/不符则关闭。

## 铁律（违反即造成不可逆破坏，务必遵守）

1. **只增不删**：唯一允许的 README 变更是**新增条目**。绝不删除、改写、重排任何已有条目。疑似重复只**跳过**。
2. **只用命令行**：所有 GitHub 操作只能用 Bash 里的 `gh` / `git`。**禁止**使用任何 GitHub 连接器或内置工具（`add_issue_comment`、`merge_pull_request` 等），即便环境里有——它们会给评论留下无法去除的 "with Claude" 归属标记。发现就忽略，一律改用 `gh api` / `gh pr` / `gh issue`。
3. **不加 reaction**：不调用任何 reaction API。
4. **去签名并验证**：通过 API 发的评论，GitHub 会自动在正文末尾追加 "Generated by Claude Code" 签名。每次发评论后：捕获评论 ID → `PATCH` 覆写为干净正文 → 再 `GET` 该评论用 `jq -r .body` 确认不含 "Generated by"；仍有就重试，最多 3 次。**未验证通过不算完成。**
5. 认证走仓库所有者的 PAT（Routine/Action 初始化时 `gh auth login --with-token`）。

设 `REPO=DevEloLin/chinese-indie-vibe-coding`。

---

## 预检：是否有新内容

```bash
REPO=DevEloLin/chinese-indie-vibe-coding
COUNT_ISSUES=$(gh api "repos/$REPO/issues?state=open&labels=投稿&per_page=50" | jq '[.[] | select(.pull_request == null)] | length')
COUNT_PRS=$(gh api "repos/$REPO/pulls?state=open&per_page=50" | jq 'length')
echo "待处理投稿 Issue: $COUNT_ISSUES  待处理 PR: $COUNT_PRS"
```

两个都为 0 → 输出「无新投稿，本次结束」并**停止**，不再执行后续步骤。

## 一、处理带「投稿」标签的 Issue

```bash
gh api "repos/$REPO/issues?state=open&labels=投稿&per_page=50" | jq '[.[] | select(.pull_request == null)]'
```

对每个 Issue，从表单正文（`### 标题\n\n值` 结构）提取字段：产品名、产品链接、一句话介绍、AI 工具（可选）、开源仓库（可选）、当前状态、是否本人。

### 1) 校验（任一不过 → 视为不符，见「关闭」）

- **链接存活**：`curl -sL -o /dev/null -w "%{http_code}" --max-time 12 "<URL>"`。返回 000 或 ≥400 视为打不开 → 不符（可在关闭评论里请其修正链接后重投）。
- **品类符合**：必须是能用的产品（网站/App/小程序/桌面/插件）。**拒**教程、课程、资源合集（awesome 类）、纯代码库/框架、纯 API/开发者工具。
- **去重（归一化后精确匹配）**：把 URL 归一化——转小写、去协议 `https?://`、去开头 `www.`、去结尾 `/`——再在全部 README 里查：
  ```bash
  norm() { echo "$1" | tr 'A-Z' 'a-z' | sed -E 's#^https?://##; s#^www\.##; s#/+$##'; }
  N=$(norm "<URL>")
  grep -rniF "$N" README.md 2>/dev/null
  ```
  用 `grep -F`（固定字符串，非正则）。命中 → 已存在，**跳过**，不动 README，在 Issue 里友好告知已收录并关闭。**严禁**用产品名/描述关键词判重（描述里提到某工具 ≠ 已收录）。

### 2) 有效 → 新增条目

确定状态标记：在运营 `:white_check_mark:` / 开发中·公测 `:clock8:` / 已关闭 `:x:`。
组装介绍（含可选的 🛠 工具、`[仓库](url)`）。目标小节：**当前「YYYY 年 M 月」节 → 提交者 `#### [名字](主页)` 子节**；没有就新增，放在「作品清单」**最上面**（倒序）。用 Read 看清结构后 Edit 精确插入，**只新增一行/一节**。改完 `git add README.md && git commit -m "add: <产品名>（投稿 #<number>）" && git push`。

然后在**该 Issue**（不是别处）发感谢并关闭（记得去签名+验证）：
```bash
CLEAN="@<提交者> 感谢投稿，已收录到清单 🎉"
RESP=$(gh api "repos/$REPO/issues/<number>/comments" -X POST -f body="$CLEAN")
CID=$(echo "$RESP" | jq -r '.id')
gh api --method PATCH "repos/$REPO/issues/comments/$CID" -f body="$CLEAN" >/dev/null
gh api "repos/$REPO/issues/comments/$CID" | jq -r '.body' | grep -q "Generated by" && echo "签名未去掉，需重试" || echo "ok"
gh issue close <number> --repo "$REPO"
```

### 3) 不符（垃圾广告/无关/信息不清/链接打不开/品类不符）→ 关闭

信息不清或链接坏可留一句友好说明再关；纯垃圾直接关：`gh issue close <number> --repo "$REPO"`。

## 二、处理开放的 PR

```bash
gh api "repos/$REPO/pulls?state=open&per_page=50"
```

对每个 PR：
1. 取 diff，确认**只新增条目、未删改已有**：`gh pr diff <number> --repo "$REPO"`。有删除/改写已有行 → 不合并，评论请其改为纯新增。
2. 对新增条目做与上面相同的校验（链接存活、品类、归一化去重）。
3. 全通过 → 合并（`gh pr merge <number> --repo "$REPO" --squash`）并感谢（同样去签名+验证）。
4. 明显垃圾/不符 → 关闭并说明。
